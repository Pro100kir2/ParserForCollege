# Парсер документов РПД — Архитектурный документ

## 1. Архитектура

Система построена по **слоистой архитектуре** с чётким разделением ответственности:

```
┌─────────────────────────────────────────────────────────┐
│                    Слой представления                   │
├──────────────────────┬──────────────────────────────────┤
│   FastAPI (Web API)  │         Интерфейс CLI            │
└──────────┬───────────┴──────────────────┬───────────────┘
           │                              │
           └──────────────┬───────────────┘
                          │
┌─────────────────────────────────────────────────────────┐
│          Слой прикладных сервисов                       │
│         (ParsingService — оркестрация процесса)         │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────┴───────────────────────────────────┐
│                    Доменный слой                         │
├──────────────┬──────────────┬──────────────┬─────────────┤
│   Чтение     │   Извлечение │  Нормализация│  Дедупли-   │
│  документов  │   секций     │    текста    │  катор      │
└──────────────┴──────────────┴──────────────┴─────────────┘
                       │
┌──────────────────────┴───────────────────────────────────┐
│                   Слой экспорта                          │
│              (ExcelExporter)                             │
└──────────────────────────────────────────────────────────┘
```

### Ключевые принципы проектирования

1. **Разделение ответственности**: Каждый слой имеет единственную обязанность
2. **Дизайн на основе интерфейсов**: Все компоненты реализуют абстрактные интерфейсы
3. **Внедрение зависимостей**: Компоненты внедряются в сервисы
4. **Управление через конфигурацию**: Псевдонимы секций и настройки вынесены наружу
5. **Устойчивость к ошибкам**: Система продолжает работу при сбоях отдельных документов
6. **Повторное использование логики**: CLI и API используют одни и те же прикладные сервисы

## 3. Основные классы и интерфейсы

### Доменные модели (`src/domain/models.py`)

```python
SectionType(Enum)
    - MAIN_LITERATURE
    - ADDITIONAL_LITERATURE
    - MATERIAL_RESOURCES

SectionContent(DataClass)
    - section_type: SectionType
    - entries: List[str]
    - raw_text: Optional[str]

Document(DataClass)
    - file_path: str
    - file_name: str
    - sections: Dict[SectionType, SectionContent]
    - parse_errors: List[str]

ParseResult(DataClass)
    - main_literature: List[str]
    - additional_literature: List[str]
    - material_resources: List[str]
    - processed_files: int
    - failed_files: int
    - failed_file_list: List[str]

ExportConfig(DataClass)
    - single_workbook: bool
    - output_dir: str
    - filename_main: str
    - filename_additional: str
    - filename_material: str
    - combined_filename: str
```

### Интерфейсы (`src/domain/interfaces.py`)

```python
DocumentReader(ABC)
    - read(file_path: str) -> Document
    - supports_format(file_path: str) -> bool

SectionExtractor(ABC)
    - extract_sections(document: Document) -> Document
    - get_section_aliases() -> Dict[SectionType, List[str]]

TextNormalizer(ABC)
    - normalize(text: str) -> str

Deduplicator(ABC)
    - deduplicate(items: List[str]) -> List[str]

Exporter(ABC)
    - export(result: ParseResult, config: ExportConfig) -> List[str]
```

### Ключевые реализации

#### Читатели документов

- **DocxReader**: Прямое чтение DOCX с помощью python-docx
- **DocReader**: Конвертация DOC через pypandoc, затем делегирование DocxReader
- **ReaderFactory**: Паттерн Фабрика для выбора подходящего читателя по расширению файла

#### Извлекатель секций

- **SectionExtractor**: Извлечение секций на основе регулярных выражений и настраиваемых псевдонимов
  - Определяет заголовки секций из конфигурации
  - Извлекает содержимое между секциями
  - Разбирает записи внутри секции

#### Процессоры

- **TextNormalizer**: Лёгкая нормализация текста
  - Обрезает пробелы
  - Сворачивает множественные пробелы
  - Удаляет завершающую пунктуацию
  - Нормализует окончания строк

- **Deduplicator**: Удаление дубликатов без учёта регистра
  - Нормализует элементы перед сравнением
  - Сохраняет оригинальный текст первого вхождения
  - Отслеживает статистику удалений

#### Экспортёр

- **ExcelExporter**: Гибкий экспорт в Excel
  - Один файл с несколькими листами
  - Несколько отдельных файлов
  - Настраивается через `ExportConfig`

#### Прикладной сервис

- **ParsingService**: Главный слой оркестрации
  - Читает документы из директории
  - Извлекает секции
  - Нормализует и дедуплицирует данные
  - Выполняет экспорт результатов
  - Грациозно обрабатывает ошибки

## 4. Процесс обработки (Workflow)

```
┌─────────────────────────────────────────────────────────┐
│  1. Вход: Путь к директории с файлами .docx/.doc        │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  2. Обнаружение документов                               │
│     - Сканирование директории на поддерживаемые файлы   │
│     - Фильтрация по расширению (.docx, .doc)            │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  3. Чтение документов (для каждого файла)                │
│     - ReaderFactory выбирает подходящий читатель         │
│     - DocxReader: Прямое чтение                          │
│     - DocReader: Конвертация в DOCX, затем чтение       │
│     - Сохранение сырого текста в объекте Document       │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  4. Извлечение секций (для каждого документа)            │
│     - Поиск заголовков секций по конфигурации            │
│     - Извлечение содержимого между секциями             │
│     - Разбор отдельных записей                           │
│     - Сохранение в Document.sections                     │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  5. Сбор данных (по всем документам)                    │
│     - Агрегация записей по типу секций                   │
│     - Формирование обобщённых списков                    │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  6. Нормализация                                          │
│     - Обрезка пробелов                                    │
│     - Сжатие множественных пробелов                      │
│     - Удаление завершающей пунктуации                    │
│     - Нормализация концов строк                          │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  7. Дедупликация                                          │
│     - Сравнение без учёта регистра                        │
│     - Удаление дубликатов в пределах каждой категории    │
│     - Сохранение оригинального текста первого вхождения  │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  8. Экспорт                                               │
│     - Один файл-книга или несколько отдельных файлов     │
│     - Формат Excel с оформленными листами                │
│     - Возврат списка созданных файлов                    │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│  9. Выход: Файлы Excel + Статистика                      │
│     - Количество обработанных файлов                     │
│     - Количество файлов с ошибками                       │
│     - Список файлов с ошибками                           │
│     - Количество записей по категориям                   │
└─────────────────────────────────────────────────────────┘
```

## 5. API

### Endpoints FastAPI

#### POST /parse

**Запрос:**
```json
{
  "input_path": "/path/to/documents",
  "output_path": "/path/to/output"  // необязательно
}
```

**Ответ (Успех):**
```json
{
  "status": "completed",
  "output_files": [
    "/path/to/output/rpd_consolidated.docx"
  ],
  "statistics": {
    "processed_files": 10,
    "failed_files": 1,
    "failed_file_list": ["document1.doc"],
    "main_literature_count": 45,
    "additional_literature_count": 32,
    "material_resources_count": 18
  }
}
```

**Ответ (Ошибка):**
```json
{
  "status": "error",
  "error": "Input path does not exist: /path/to/documents"
}
```

#### GET /

**Ответ:**
```json
{
  "message": "RPD Document Parser API",
  "version": "1.0.0",
  "endpoints": {
    "parse": "/parse",
    "health": "/health"
  }
}
```

#### GET /health

**Ответ:**
```json
{
  "status": "healthy"
}
```

### Принципы дизайна API

1. **RESTful**: Стандартные HTTP-методы и коды статусов
2. **JSON**: Запросы и ответы в формате JSON
3. **Валидация**: Pydantic-модели для проверки запросов
4. **Обработка ошибок**: Единообразный формат ответов об ошибках
5. **Документация**: Автогенерация через FastAPI OpenAPI
6. **Готовность к фронтенду**: Разработан с учётом будущего подключения веб-интерфейса

## 6. Дизайн CLI

### Структура команды

```bash
python main.py --input <директория> [--output <директория>] [--single-workbook] [--verbose]
```

### Аргументы

- `--input` / `-i` (обязательный): Путь к директории с документами РПД
- `--output` / `-o` (необязательный): Директория для сохранения Excel-файлов (по умолчанию: `./output`)
- `--single-workbook` / `-s` (необязательный): Экспорт в один файл-книгу с несколькими листами
- `--verbose` / `-v`(необязательный): Включить подробное логирование

### Примеры использования

```bash
# Базовое использование
python main.py --input ./rpd

# С указанием директории вывода
python main.py --input ./rpd --output ./result

# Экспорт в один файл
python main.py --input ./rpd --single-workbook

# Подробное логирование
python main.py --input ./rpd --verbose
```

5. **Кросс-платформенность**: Работает на Windows, macOS и Linux

| pytest       | Тестирование                      | При расширении набора тестов |
| Docker       | Развёртывание                     | При необходимости контейнеризации |

## 9. Риски и стратегии их минимизации

### Технические риски

#### Риск 1: Вариативность формата DOC

**Описание**: Файлы DOC могут иметь разную внутреннюю структуру, которую инструменты конвертации не всегда корректно обрабатывают.

**Стратегии минимизации**:
- Использование нескольких стратегий конвертации (pypandoc с разными бэкендами)
- Резервные инструкции по ручной конвертации
- Чёткие сообщения об ошибках с указанием проблемных файлов
- Подробное логирование процесса конвертации

**Статус**: Реализовано через pypandoc, предусмотрена обработка fallback

---

#### Риск 2: Вариации названий секций

**Описание**: Названия секций могут незначительно отличаться от заданных в конфигурации.

**Стратегии минимизации**:
- Настраиваемые псевдонимы в настройках
- В будущем: добавление нечёткого сопоставления (fuzzy matching)
- В будущем: ML-распознавание секций
- Логирование нераспознанных секций для дальнейшего анализа

**Статус**: Настраиваемые псевдонимы реализованы, fuzzy matching — в планах

---

#### Риск 3: Обработка больших файлов

**Описание**: Очень большие документы могут вызывать проблемы с памятью.

**Стратегии минимизации**:
- Потоковое чтение документов (будущая доработка)
- Экономичное использование памяти при парсинге
- Предупреждения о большом размере файлов
- Обработка по частям для очень больших директорий

**Статус**: Сейчас документ загружается целиком; потоковая обработка — в планах

---

#### Риск 4: Проблемы с кодировками

**Описание**: Документы могут иметь нестандартные кодировки.

**Стратегии минимизации**:
- Сквозная работа с UTF-8
- Автоопределение кодировки для DOC-файлов
- Резервные стратегии кодирования
- Чёткие сообщения об ошибках кодирования

**Статус**: Обработка UTF-8 реализована, автоопределение кодировки включено в слой конвертации

---

#### Риск 5: Одновременный доступ

**Описание**: Несколько запросов API могут пытаться одновременно обработать одну и ту же директорию.

**Стратегии минимизации**:
- Блокировка файлов при записи (в будущем)
- Уникальные директории вывода для каждого запроса
- Работа с временными файлами
- Отслеживание запросов по ID

**Статус**: Уникальные директории вывода для каждого запроса уже реализованы

---

### Операционные риски

#### Риск 6: Отсутствие зависимостей на целевой системе

**Описание**: pypandoc требует установленного LibreOffice или pandoc.

**Стратегии минимизации**:
- Подробные инструкции по установке в README
- Проверка зависимостей при запуске
- Graceful degradation (пропуск DOC-файлов при отсутствии конвертера)
- Docker-контейнер со всеми зависимостями (в будущем)

**Статус**: Инструкции по установке предоставлены, graceful degradation реализован

---

#### Риск 7: Повреждённые документы

**Описание**: Испорченные или некорректные документы могут привести к падению парсера.

**Стратегии минимизации**:
- Блоки try-catch вокруг всех операций с документами
- Изоляция ошибок отдельных документов
- Подробное логирование ошибок
- Продолжение обработки остальных документов

**Статус**: Комплексная обработка ошибок реализована

---

#### Риск 8: Производительность при больших объёмах

**Описание**: Обработка большого количества документов может быть медленной.

**Стратегии минимизации**:
- Сейчас: последовательная обработка
- В будущем: асинхронная / параллельная обработка
- В будущем: отчёт о прогрессе
- В будущем: кэширование ранее обработанных документов

**Статус**: Последовательная обработка реализована, асинхронность — в планах

- Конфигурация с поддержкой версий (в будущем)

**Статус**: Настраиваемый дизайн реализован

**Стратегии минимизации**:

## 10. Развёртывание

### Разработка

```bash
# Установка зависимостей
pip install -r requirements.txt

# Запуск CLI
python main.py --input ./test_documents

# Запуск API
uvicorn src.api.main:app --reload
```

```bash
# Установка зависимостей
pip install -r requirements.txt

# Запуск API в продакшен-режиме
uvicorn src.api.main:app --host 0.0.0.0 --port 8000 --workers 4
```
